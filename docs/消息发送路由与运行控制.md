# 消息发送路由与运行控制

## 当前状态
消息发送由 `useComposerSend` 和 `useMessageActions` 协作完成。前者负责本地输入决策、slash 命令、steer 队列和 send event，后者监听 `agent:send-message` 并创建 query stream session。

## 核心职责
- 解析目标 chat、agentKey、teamId、附件 references、运行参数和模型覆盖。
- 阻止同一会话重复并发发送，支持不同会话切换时 detach 当前 run。
- 在 streaming 状态下将普通输入转为 steer，且禁止带附件 steer。
- 管理 abort controller、requestId、runId、live session 和 timeline user message。

## 核心流程
用户触发发送后，`useComposerSend` 先处理 slash command 和 steer 场景，再派发 `agent:send-message`。`useMessageActions` 接收事件，解析 mention 和 preferred routing，创建 requestId，启动 SSE/WS executor，并将收到的事件交给 timeline handler。

## 边界与非目标
- 发送路由只提供前端路由提示；后端仍是 agent/team 解析事实源。
- 不在发送流程中解释 tool、content、plan 等事件，这些由 timeline processor 负责。
- steer、interrupt、compact、remember、learn 等命令的协议字段以后端接口为准。
